# NeonHub Admin

The AI-catalog orchestration UI for **NeonHub**, a neon-sign shop. It is a
Next.js (App Router) admin panel that a human operator uses to build, submit,
and triage AI-generated product imagery and video at catalog scale.

## What it does

The catalog is generated by crossing **ideas** with **prompts** to produce a
matrix of **tasks**. Each task is one image-generation request (a product idea
rendered through a specific prompt, aspect ratio, and model). The operator then
ships those tasks to image generation, waits for results, and curates what comes
back.

The end-to-end flow:

1. **Idea × Prompt → Task.** `Idea to Task` / `Task to Task` build the task
   matrix from ideas and prompt groups (or from the results of earlier tasks).
2. **Submit in 50-row batches.** `Submit Job` selects pending tasks and posts
   them to the external job server in batches of 50. Batching is what unlocks
   the cheaper Gemini **Batch API** path (see the honest note below).
3. **Poll.** `Jobs in progress` tracks active jobs that have no result yet.
4. **Triage.** `Photo Review`, `Product Stage`, `Product Manager`,
   `Variant Colors`, `Approve Products`, `Product Review`, and
   `Product Categories` let a human sort, approve, color-tag, and categorize the
   returned images per product.
5. **Video.** `Products to Video`, `Task to Video`, `Video Review`,
   `Video Builder`, and `Video by Product` turn approved imagery into short
   product videos and review the renders.

## Honest scope note

**The batch/cost mechanics are external. They are not in this repo.** The
Gemini "Nano Banana" Batch API integration (which submits jobs in 50-row
batches for the **-50%** batch-pricing discount), the polling/job lifecycle, and
the actual image/video generation all run in **external systems: an n8n
workflow layer and a separate job server.** They are described in the
architecture docs, not implemented here.

This repository is **only the admin UI**: it builds the task matrix, calls those
external systems over HTTP, and provides the human triage surface. Every webhook
and job-server endpoint is configured via environment variables (below) and
points at placeholder hosts out of the box.

## Architecture (this repo)

- **`app/`**: App Router pages (the screens listed above) plus `app/api/*`
  route handlers that proxy to the external n8n webhooks and job server.
- **`app/api/submit`**: server route that forwards a batch to the job server
  (`POST /create-job`).
- **`app/api/submit/tasks`**: server proxy for reading/updating tasks in n8n
  (used by the Submit screen so the browser never talks to n8n directly).
- **`components/`**, **`lib/`**, **`hooks/`**: shadcn/ui components, the demo
  auth gate, and utilities.

## Environment variables

Copy `.env.example` to `.env.local` and fill in real values.

| Variable | Purpose |
| --- | --- |
| `N8N_BASE_URL` | Base host of the n8n instance serving the content webhooks (server-side). |
| `JOB_SERVER_URL` | Job server that accepts batch jobs at `POST /create-job` (server-side). |
| `NEXT_PUBLIC_JOB_SERVER_URL` | Optional browser-exposed job-server URL; the server route falls back to `JOB_SERVER_URL`. |
| `NEXT_PUBLIC_DIRECTUS_URL` | Directus instance that stores rendered video assets. |
| `ADMIN_PASSWORD` | Password for the thin client-side demo gate in `lib/auth.ts`. Not real auth. |

## Getting started

```bash
pnpm install
cp .env.example .env.local   # then edit values
pnpm dev
```

Open http://localhost:3000.

> **Auth is a demo gate.** `lib/auth.ts` compares a single shared password and
> stores a flag in `localStorage`. It only hides the UI; it is not a security
> boundary. Put real authentication in front of any real deployment.
