# NeonHub: Storefront

Production storefront for **NeonHub**, a custom neon-sign maker. Built as a
**Next.js static export** (`output: 'export'`). Every route is pre-rendered to
plain HTML/CSS/JS at build time and served from any static host or CDN, with no
Node server in production.

This public repo is an **honest, English-translated showcase** of the real
storefront. The live site is data-driven from a headless catalog of hundreds of
products; here that catalog is replaced with a small **demo dataset** (~48
generic neon-sign products across 24 categories) that keeps the exact data
**shape**, so the SEO, sitemap and image pipelines still run end to end. Regulated
categories and all personal/business identifiers from the production site have
been removed. Product images live under `public/images/` (git-ignored) and are
git-ignored and generated by the image pipeline below; the demo catalog
references placeholder image paths.

The interesting part of this project is the **SEO and performance engineering**,
described below.

## Stack

- **Next.js 16** (App Router) + **React 19**, TypeScript
- **Static export** (`output: 'export'`, `trailingSlash: true`, `images.unoptimized`)
- **Tailwind CSS v4**, Radix UI / shadcn-style primitives
- **sharp** for the build-time WebP / watermark image pipeline
- Client analytics: Yandex Metrika + VK Pixel (env-gated)

## SEO engineering

Every surface a crawler can see is generated, typed and per-page tuned.

### Metadata
- Root `Metadata` in `app/layout.tsx`: `metadataBase`, title/description,
  keywords, canonical, robots, full **OpenGraph** + icons, and a Pinterest
  domain-verification tag (env-gated).
- **Per-product `generateMetadata`** in `app/product/[slug]/page.tsx` builds a
  unique title, description, keyword set, canonical and OG image list for each of
  the statically generated product pages (`generateStaticParams` +
  `dynamicParams = false`).
- Category pages (`app/catalog/[category]/page.tsx`) derive their own
  title/description/keywords/intro copy from the catalog.

### Structured data (JSON-LD)
- **Organization / Store** and **WebSite** on the home page and root layout.
- **Product** schema (SKU, image set, price `Offer`, brand, seller) on every
  product page.
- **BreadcrumbList** on product pages (Home → Catalog → Category → Product).
- All JSON-LD is serialized with `<` escaped to `<` to stay XSS-safe inside
  `dangerouslySetInnerHTML`.

### Sitemaps, robots & instant indexing
- `app/sitemap.ts` - typed `MetadataRoute.Sitemap`: home, catalog, info pages,
  one entry per category and per product group, with per-product `images` and
  `lastModified` taken from the catalog file mtime.
- `app/image-sitemap.xml/route.ts` - a dedicated **image sitemap** (force-static
  route handler) so product imagery is discoverable.
- `app/robots.ts` - typed robots with both sitemaps and host.
- `scripts/indexnow.mjs` - pings **IndexNow** with the full URL list (home,
  catalog, categories, all product groups) after each deploy, so search engines
  re-crawl immediately. Key and host come from env.

### Keyword model
- `public/data/keywords.json` holds common + per-category keyword pools.
- `lib/keywords.server.ts` composes deduplicated, length-trimmed keyword strings
  and SEO copy for home, category and product contexts.

### Marketing product feeds
Build-time route handlers emit ready-to-submit product feeds from the same
catalog:
- `app/yandex-products.xml` & `app/yandex-bussines.xml` (`lib/catalog-yml-feed.ts`,
  `lib/yandex-business-feed.ts`)
- `app/pinterest-feed.xml` (Pinterest catalog / RSS)
- `app/vk-feed.xml` (VK marketing feed)

## Performance engineering

### Static export
`output: 'export'` produces a fully static `out/` directory: no SSR at runtime,
trivially cacheable, deployable to any static host (see `deploy.sh`).

### Fonts
`next/font/google` (Orbitron, Inter, Tektur, Rubik Glitch) with
`display: 'swap'` and selective `preload: false` on secondary faces: no layout
shift, no render-blocking font CSS, self-hosted at build.

### Low-end-device detector
A tiny inline script in `app/layout.tsx` flags low-end clients
(`hardwareConcurrency`, `deviceMemory`, `navigator.connection.saveData` /
`effectiveType`) and adds a `low-end` class to `<html>`, letting CSS drop the
heaviest neon glow/animation work on weak devices.

### Image pipeline (build-time WebP + watermark)
- Source product images are stored as `.webp`. `lib/utils.ts#directusImage`
  resolves the right pre-generated variant: full **`-wm.webp`** (watermarked) and
  thumbnail **`-thumb-wm.webp`** versions, plus size-preview images.
- `scripts/watermark-images.mjs` runs as the **`prebuild`** step:
  `scripts/product-watermarks.mjs` uses **sharp** to tile a rotated brand
  watermark across every product image, writing cached `-wm` outputs (skips
  unchanged files via mtime, prunes orphans, bounded concurrency).
- `scripts/generate-watermark-logo-assets.mjs` prebuilds the watermark logo
  assets.
- `images.unoptimized` is set because optimization happens in this pipeline, not
  at request time, consistent with a static host.

### Caching
`public/.htaccess` sets long-lived cache headers + gzip for the catalog JSON;
the catalog mtime feeds `NEXT_PUBLIC_PRODUCTS_VERSION` (`next.config.mjs`) for
cache-busting.

## Local development

```bash
npm install
npm run dev      # http://localhost:3000
npm run build    # static export -> ./out  (runs the watermark prebuild)
```

Copy `.env.example` to `.env.local` and fill in values as needed. Analytics and
the order webhook are inert until their env vars are set.

## Deployment

`deploy.sh` builds the static export and `rsync`s `./out` to a host over
**key-based SSH** (host/user/path from env), then pings IndexNow. No credentials
are stored in the repo.

## Notes

- **Demo data**: `public/data/products.json` is a trimmed, English, USD demo
  catalog, not the production catalog.
- **Images** under `public/images/` are git-ignored and produced by the
  pipeline above (no binary product images are committed); the demo catalog
  references placeholder image paths, and the static export builds regardless.
