# Hero A: A fully AI-generated product catalog (Gemini "Nano Banana" Batch API, -50% cost)

NeonHub's entire product catalog (thousands of neon-sign images across five visual categories) is **generated by AI**, not photographed. The `admin/` app is the cockpit for that pipeline: it turns a small library of prompts and ideas into thousands of image-generation tasks, ships them to Google's Gemini image models through the **Batch API at half the interactive price**, and gives a human a fast way to triage the output into a real catalog.

> **What's in this repo vs. what's described.** The `admin/` Next.js app (task builder, batch submit, job monitor, review UIs) is here in full. The **actual Gemini Batch API request, the API key, and the polling against Google** run in an external **n8n workflow + job server**. This doc *describes* that layer; it is not shipped. What the repo *does* show is the precise batch-submit-then-poll shape, which is exactly how the Gemini Batch API works.

## Why Batch (the -50% mechanic)

Generating a product catalog is a throughput problem, not a latency problem. Nobody is waiting on any single image. Google's **Gemini Batch API** is built for exactly this: you submit many requests as one asynchronous job and get results back within a turnaround window, billed at **50% of the interactive (synchronous) rate**. For a catalog of thousands of images, routing generation through Batch instead of live calls **halves the bill** with no quality cost. That single architectural choice is the headline.

## The pipeline, stage by stage

### 1. Prompt library × ideas
A Google Sheet holds reusable **prompt templates** (across `idea`, `jewelry`, `interior`, `person`, `child`) and a pool of product **ideas**. Each template carries `{{var}}` placeholders and fixed reference-image slots.

### 2. Task generation: the cartesian product
In `admin/app/idea-to-task`, the operator multi-selects *ideas × prompts × N duplicates*. The app:
- computes the **cartesian product** of the selection,
- substitutes each template's `{{var}}` tags from the idea's columns (`resolvePrompt()`),
- chains up to **10 reference images** per row,
- appends the rows (in batches of 500) to the `NanoBananaTasks` sheet.

These rows are **not yet sent to Gemini**. They are a queue. A refine variant (`task-to-task`) feeds a previous generation's output image back in as the reference, to iterate on a result.

### 3. Batch submit: the cost core
In `admin/app/submit`, un-submitted rows (`!job_id && !result_img`) are sliced into **batches of 50** and posted to a single `create-job` endpoint. The external **job server** assembles each batch into one **Gemini Batch API** job and returns a `job_id` (`job_<epoch-ms>`), which is written back onto every row in the batch.

```
selected rows ─► chunk(50) ─► POST create-job ─► Gemini Batch API (-50%)
                                     │
                                     └─► job_id written back to each row
```

### 4. Poll & monitor
`admin/app/jobs` lists in-flight rows (`job_id && !result_img`) grouped by job and refreshes the **elapsed-time** display every 10s; the Refresh button re-pulls job status. As Gemini completes a job, the external workflow writes each generated image's **Directus asset URL** into the row's `result_img`.

### 5. Human triage → catalog
The review UIs (`task-sorter`, `product-stage`, `variant-colors`, `products`, ...) let a human mark each image **cool/defect**, choose the **hero** image per product (`is_main`), assign colors/categories/sizes, and approve. An optional video stage turns approved images into promo clips.

## The takeaway

The interesting engineering here is **systems design, not a single model call**: a spreadsheet-backed task queue, a fan-out into fixed-size asynchronous batches, an idempotent `job_id` write-back, a polling monitor, and a human-in-the-loop triage funnel, all arranged so that generating a catalog of thousands of images is **cheap (Batch -50%), resumable, and reviewable**.
